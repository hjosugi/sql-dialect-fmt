# sql-dialect-fmt: Prior-Art Research & Distilled Lessons

## Executive summary

A best-in-class Snowflake SQL formatter + highlighter is an unoccupied niche: every existing SQL tool fails at least one of *{lossless comment-preserving CST}* × *{deep Snowflake grammar}* × *{Prettier-style Doc-IR single-pass pretty-printer}* × *{Rust speed}*. The winning blueprint is the one Biome and Ruff already proved for JS/Python: a hand-written, error-resilient recursive-descent parser that emits a flat **event stream** which builds a **rowan** lossless green/red CST (à la rust-analyzer); a thin typed AST layer; a generic **Doc-IR formatter engine** (`FormatElement` with `group`/`line`/`indent`/`best_fitting`) descended from Prettier → `rome_formatter` → `biome_formatter`/`ruff_formatter`; per-node **leading/trailing/dangling** comment attachment; trailing comments via `line_suffix`; embedded `$$ … $$` bodies handed to `biome_js_formatter` and re-indented via a `markAsRoot`/`dedentToRoot`-style root; highlighting staged **LSP semantic tokens → TextMate → tree-sitter**; and conformance proven by snapshot (`insta`) + idempotency + a Ruff-style similarity-score corpus harness. sql-dialect-fmt's lexer is done, so the immediate work is parser-events + CST + the Doc engine.

## Per-area recommendations

### 1. CST construction — rowan builder + parser events
**Use rowan (or copy its green/red split) and decouple the parser from tree-building via an event stream.** Green nodes are immutable, position-free, deduplicated (so trivia tokens are first-class and round-trip byte-exact); red nodes add parents/offsets lazily — ideal for a lossless formatter CST. *Rationale:* this is rust-analyzer's proven design and what Biome forked. ([rowan green/red](https://github.com/rust-analyzer/rowan), [RA syntax book](https://rust-analyzer.github.io/book/contributing/syntax.html))
- Define one flat `#[repr(u16)] SyntaxKind` enum mixing token + node kinds; implement rowan's `Language` for a `SnowLang {}` marker; add a `T![,]`/`T![select]` macro. ([RA syntax_kind](https://github.com/rust-lang/rust-analyzer/blob/master/crates/parser/src/syntax_kind/generated.rs))
- Parser pushes `Vec<Event>` (`Start{kind,forward_parent}` / `Token` / `Finish` / `Error`); a separate pass feeds events into `GreenNodeBuilder`, **attaching whitespace/comments there** so the grammar stays clean but the tree stays lossless. ([RA event.rs](https://github.com/rust-lang/rust-analyzer/blob/master/crates/parser/src/event.rs))
- Use the `Marker`/`CompletedMarker` API with a `DropBomb`: `m=p.start()`, `m.complete(p,KIND)`, `m.abandon(p)` (backtrack), and `cm.precede(p)` for left-associative SQL (`a.b.c`, `t1 JOIN t2 JOIN t3`, `a OR b OR c`). `GreenNodeBuilder::checkpoint`/`start_node_at` is the equivalent retroactive-wrap trick. ([RA parser.rs](https://github.com/rust-lang/rust-analyzer/blob/master/crates/parser/src/parser.rs), [GreenNodeBuilder](https://docs.rs/rowan/latest/rowan/struct.GreenNodeBuilder.html))
- Make **"parsing never fails"** the law: return `(SyntaxNode, Vec<SyntaxError>)`, wrap stray tokens in ERROR nodes, recover with `TokenSet` FOLLOW sets (clause-start keywords / `;`), and guard loops with a fuel counter. The directly applicable blueprint is matklad's [Resilient LL Parsing](https://matklad.github.io/2023/05/21/resilient-ll-parsing-tutorial.html); use [Pratt parsing](https://matklad.github.io/2020/04/13/simple-but-powerful-pratt-parsing.html) for SQL's many precedence tiers.
- Add thin typed AST newtypes (`struct SelectClause{syntax:SyntaxNode}` with `cast`/`syntax`/`support::child`). Defer full ungrammar codegen until the grammar settles. ([RA AstNode](https://github.com/rust-lang/rust-analyzer/blob/master/crates/syntax/src/ast/generated/nodes.rs))
- Keep an **official-spec-derived parser/conformance lane** open. The `postgresql-cst-parser` story is the useful pattern: if an upstream grammar or machine-readable syntax inventory exists, generate as much of the CST parser surface as possible in Pure Rust so new syntax support mostly becomes formatter-rule work. Snowflake does not expose a PostgreSQL-style `gram.y`, so the practical near-term version is a docs/examples miner that emits coverage fixtures, keyword deltas, statement skeletons, and parser-gap reports. Do not swap the handwritten parser out until a generated path can preserve trivia, recover from partial input, remain compact, and compile cleanly to Wasm.

### 2. Formatter IR — roll a Doc engine modeled on biome/ruff
**Build a generic `snow_formatter` engine crate (don't depend on `biome_formatter` directly) whose IR copies `FormatElement`, and keep all Snowflake knowledge in a `snow_formatter_sql` layer.** The proven lineage is Prettier's Doc → `rome_formatter` → `{biome_formatter, ruff_formatter}`; Ruff explicitly "adapted Biome's excellent printer." *Rationale:* a measure-to-width Wadler/Prettier printer is the only model that can keep short `SELECT`s on one line yet wrap long ones at the width limit — which token-stream tools (sql-formatter) and reflow-by-input tools (Topiary, sqlfluff) cannot. ([ruff_formatter lib.rs](https://github.com/astral-sh/ruff/blob/main/crates/ruff_formatter/src/lib.rs), [Ruff blog](https://astral.sh/blog/the-ruff-formatter), [Biome formatter](https://deepwiki.com/biomejs/biome/6.2-formatter-implementation))
- **IR variants:** `Text` / `SourceCodeSlice` (zero-copy), `Line(LineMode::{Soft,Hard,Empty,SoftOrSpace})`, `Tag(StartGroup/EndGroup/StartIndent/EndIndent…)`, `LineSuffix` + `LineSuffixBoundary`, `BestFitting` (try layouts, pick first that fits = Prettier `conditionalGroup`), `Interned`, `ExpandParent`. ([ruff format_element.rs](https://github.com/astral-sh/ruff/blob/main/crates/ruff_formatter/src/format_element.rs), [Prettier commands.md](https://github.com/prettier/prettier/blob/main/commands.md))
- **Builders:** `group`, `indent`/`dedent`, `block_indent`/`soft_block_indent`, the four line breaks, `line_suffix`, `if_group_breaks`, `best_fitting`. ([ruff builders.rs](https://github.com/astral-sh/ruff/blob/main/crates/ruff_formatter/src/builders.rs))
- **Traits:** buffer-based `Format<Context>` + `FormatRule<T,Context>` + `FormatNodeRule<N>` (so AST types from the parser crate are formatted without orphan-impl pain), with `write!`/`format_args!`/`format!` macros. ([Biome traits](https://deepwiki.com/biomejs/biome/6.2-formatter-implementation))
- Keep a dprint-style column-aware `Condition`/`Info` as a *narrow* escape hatch only for alias/`ON`-predicate alignment — not the default. ([dprint](https://github.com/dprint/dprint))
- **Magic trailing comma** (highest-leverage feature): a user trailing comma in a `SELECT` list / `IN (...)` / `VALUES` / arg list forces the exploded layout (inject `expand_parent`); its absence permits collapsing. This is the zig-fmt "steering" signal and survives round-trips. ([Ruff black deviations](https://docs.astral.sh/ruff/formatter/black/), [zig fmt](https://matklad.github.io/2026/05/08/steering-zig-fmt.html))

### 3. Comment attachment — leading / trailing / dangling, per node
**Attach comments per-node (not per-token) into three buckets and emit trailing comments via `line_suffix`.** Both Biome (`place_comment` → `CommentPlacement`, `DecoratedComment`, `CommentStyle`) and Ruff (`comments/placement.rs`) use exactly this; Ruff notes per-node attachment "greatly reduces the combinations… but turns out to be sufficiently precise." *Rationale:* fewer positions = tractable + idempotent. ([Biome comments](https://deepwiki.com/biomejs/biome/6.2-formatter-implementation), [Ruff comments/mod.rs](https://github.com/astral-sh/ruff/blob/main/crates/ruff_python_formatter/src/comments/mod.rs))
- Algorithm (Prettier `decorateComment`/`attachComments`): for each comment find `enclosingNode`, `precedingNode`, `followingNode`; **followingNode ⇒ leading**, **precedingNode ⇒ trailing**, **neither ⇒ dangling** (belongs to enclosing, e.g. a lone comment inside `( )`). Use `isOwnLineComment`/`isEndOfLineComment` (newline-before/after) and `breakTies` for the ambiguous middle case. ([Prettier attach.js](https://github.com/prettier/prettier/blob/main/src/main/comments/attach.js))
- Add SQL-specific override handlers (like Ruff's `handle_*` heuristics): comments after `SELECT`, between `WHEN`/`THEN`, before a JOIN's `ON`, inside `( )`. ([Ruff placement.rs](https://github.com/astral-sh/ruff/blob/main/crates/ruff_python_formatter/src/comments/placement.rs))
- **Exclude directive comments from width** (Ruff excludes `# noqa`/`# type:`): a long trailing `-- noqa`/`-- sql-dialect-fmt:` shouldn't trigger wrapping. Invariant: re-formatting output is byte-identical and never produces invalid SQL — test via stability-check. ([Ruff black deviations](https://docs.astral.sh/ruff/formatter/black/))

### 4. Embedded-language formatting — call `biome_js_formatter`, re-indent to a root
**Reuse `biome_js_formatter` for `$$ … $$` JS bodies; format the body standalone then re-indent the whole block to its placement column.** Biome already ships embedded CSS-in-JS snippet formatting, so the same crate-level entry exists for JS. *Rationale:* don't re-implement a JS formatter. ([Biome embedded CSS-in-JS](https://github.com/biomejs/biome/commit/bc0e8b47a276efabb0b76169d13dfc9d5325953f), [Biome v2.4 embedded snippets](https://biomejs.dev/blog/biome-v2-4/), [issue #3334](https://github.com/biomejs/biome/issues/3334))
- Re-indentation: Prettier's pattern is `markAsRoot` to set a new indentation root at the embed site, `dedentToRoot`/`literalline` so the embedded doc indents relative to that root, and `lineSuffixBoundary` so comments "shouldn't be able to leave the code part." Mirror this when splicing the JS doc into the SQL doc. ([Prettier commands.md markAsRoot/dedentToRoot](https://github.com/prettier/prettier/blob/main/commands.md))
- Treat the dollar-quoted body as an injection/leaf for highlighting (`@injection.content` + `@injection.language` from the `LANGUAGE` clause), exactly as tree-sitter handles SQL-in-JS. ([tree-sitter syntax highlighting / injections](https://tree-sitter.github.io/tree-sitter/3-syntax-highlighting.html))
- Same path extends later to Python/Java/Scala UDF bodies. Always keep an "as-is / verbatim" fallback for unparseable bodies so the formatter is total.

### 5. Highlighting — staged: LSP semantic tokens → TextMate → tree-sitter
**Ship in this order.** *Rationale:* you already have the parser, so the most accurate route is free; tree-sitter is a separate, maintenance-heavy grammar whose payoff is editor-ecosystem reach, not formatter quality.
1. **LSP semantic tokens** from your CST (`textDocument/semanticTokens`) — most accurate, zero new grammar, works in VS Code/Neovim/Zed/Helix. ([tree-sitter highlighting overview for comparison](https://tree-sitter.github.io/tree-sitter/3-syntax-highlighting.html))
2. **TextMate grammar** — cheap universal fallback for VS Code/web/GitHub-style highlighting where your LSP isn't running.
3. **tree-sitter grammar + `highlights.scm`/`injections.scm`** only when you want native Neovim/Helix/Zed and GitHub.com highlighting + code-nav (the latter also needs a Linguist entry and a crates.io-published parser). Accept this is a *second* parser, separate from the formatter's. ([tree-sitter code navigation](https://tree-sitter.github.io/tree-sitter/4-code-navigation.html))
- Whatever the source, name captures with Neovim's dotted vocabulary (`@keyword`, `@keyword.operator`, `@string`, `@string.escape`, `@number`, `@function.call`, `@type.builtin`, `@operator`, `@comment`, `@punctuation.bracket`) so themes and a future `highlights.scm` are drop-in across editors. ([Neovim treesitter highlight groups](https://neovim.io/doc/user/treesitter.html), [DerekStride/tree-sitter-sql queries](https://github.com/DerekStride/tree-sitter-sql))
- **Do not** adopt Topiary as the formatter engine: its softlines resolve purely on whether a node spans >1 input line (no width budget), and it can squash tokens — borrow only its capture set as a decision checklist. ([Topiary](https://github.com/tweag/topiary))

### 6. Testing — snapshot + idempotency + round-trip + conformance corpus
**Adopt Ruff's whole test stack.** ([Ruff formatter CONTRIBUTING](https://github.com/astral-sh/ruff/blob/main/crates/ruff_python_formatter/CONTRIBUTING.md))
- **Snapshot tests with `insta`** over `resources/test/fixtures` (lift sqlparser-rs `tests/sqlparser_snowflake.rs` SQL strings, Apache-2.0, as inputs). ([sqlparser snowflake tests](https://github.com/apache/datafusion-sqlparser-rs/blob/main/tests/sqlparser_snowflake.rs))
- **Round-trip / lossless check:** concatenating all CST token texts must reproduce the source byte-for-byte (rowan guarantees this if you build it right). ([RA syntax book](https://rust-analyzer.github.io/book/contributing/syntax.html))
- **Idempotency `--stability-check`:** format twice, assert byte-equality; also catches panics and invalid-SQL output (sqlfluff's looped-fix model fails exactly here — non-idempotency [#2134](https://github.com/sqlfluff/sqlfluff/issues/2134)). ([Ruff stability](https://github.com/astral-sh/ruff/blob/main/crates/ruff_python_formatter/CONTRIBUTING.md))
- **Conformance harness** `snow_dev format-dev`: run over a corpus of real Snowflake SQL (dbt projects, Snowflake docs/sample queries) and report a per-file **similarity = neutral / (neutral + removed)** diff score as a CI gate. Ruff reports >99.9% line-identical vs Black this way. ([Ruff format-dev / similarity](https://github.com/astral-sh/ruff/blob/main/crates/ruff_python_formatter/CONTRIBUTING.md))
- Add a small **fuzz** target (random/mutated SQL) asserting the parser never panics and output is always valid + idempotent.

### 7. Performance — interning, single pass, parallelism, incremental
- **Arena/interning:** share one `NodeCache` (FxHash) across the `GreenNodeBuilder` to dedup tokens; intern identifiers. rowan's structural sharing makes this cheap. ([rowan node_cache](https://github.com/rust-analyzer/rowan/blob/master/src/green/node_cache.rs))
- **Single pass + zero-copy:** emit `SourceCodeSlice` instead of copying text; the Doc engine is one CST walk + one printer pass (vs sqlfluff's convergence loop, default 10 iterations). ([Ruff format_element.rs](https://github.com/astral-sh/ruff/blob/main/crates/ruff_formatter/src/format_element.rs), [sqlfluff loop-limit #5325](https://github.com/sqlfluff/sqlfluff/issues/5325))
- **Parallelism:** format many files across threads (rayon); each file is independent. This + Rust is the headline win over Python sqlfluff (minutes-to-hours on large files; it is itself porting to Rust). ([sqlfluff 4.0 Rust](https://github.com/sqlfluff/sqlfluff/releases/tag/4.0.0))
- **Incremental for LSP:** rowan green-tree reuse enables patching a prior tree for fast reparse on keystroke — keep the event/builder path edit-friendly even though the v1 formatter is whole-file. ([RA architecture](https://rust-analyzer.github.io/book/contributing/architecture.html))
- **Pure Rust / Wasm pressure:** keep the formatter/LSP parser stack independent from C-based editor grammars. Tree-sitter can remain a parallel ecosystem asset, but the browser, Chrome extension, and future web playground should depend on the Pure Rust CST path so packaging stays small and predictable.

## Apply-now vs later checklist

**Now (lexer done; parser/formatter not started):**
1. Define `SyntaxKind` (tokens + nodes) + `SnowLang` rowan `Language` + `T!` macro.
2. Build the parser as an **event emitter** (`Marker`/`complete`/`abandon`/`precede`, `TokenSet` recovery, fuel guard, "never fails"), feeding a `GreenNodeBuilder` that attaches trivia. Verify byte-exact round-trip.
3. Stand up the generic **Doc-IR engine** (`FormatElement`, `Format`/`FormatRule`, `group`/`indent`/`line`/`best_fitting`, `line_suffix`) before any SQL rules.
4. Implement comment attachment (leading/trailing/dangling) + the **magic trailing comma**.
5. Wire `insta` snapshots + `--stability-check` from day one; seed fixtures from sqlparser-rs Snowflake tests.
6. Ship **opinionated, near-zero-config** (`line-length`, maybe `keyword-case`).

**Later:**
7. Grow Snowflake grammar coverage using **sqlfluff `dialect_snowflake.py`** as the checklist (QUALIFY, semi-structured `:`/`[]`, PIVOT/UNPIVOT, FLATTEN, MATCH_RECOGNIZE, COPY INTO, CREATE WAREHOUSE/STREAM/TASK).
8. Embedded `$$…$$` JS via `biome_js_formatter` + root re-indent; verbatim fallback.
9. LSP **semantic-token** highlighting; then a **TextMate** grammar.
10. `snow_dev format-dev` **similarity-score** conformance gate over a real corpus.
11. Optional: official-spec-derived parser/conformance generator; ungrammar codegen; dbt/Jinja templating via a sqlfluff-style literal-anchored slice map; a tree-sitter grammar for Neovim/GitHub.
12. Optional LSP lint lane: rule diagnostics that do not require database connectivity first (oversized `IN` lists, suspicious nullable joins when schema metadata is provided later, object-reference checks when a catalog snapshot is available).
